SeatSavvy

A cloud-native, microservices-based event seat reservation platform built with polyglot architecture — Python, Java, Node.js, and React — deployed on Kubernetes with full observability through Prometheus and Grafana.

☁️ Kubernetes 🐍 Python · FastAPI ☕ Java · Spring Boot ⬡ Node.js · Express ⚛️ React · Vite 🐇 RabbitMQ 📊 Prometheus · Grafana

Product Overview

SeatSavvy is a full-stack event ticketing system that enables users to browse events, select seats on an interactive grid, process payments, and receive digital e-tickets. The system is designed with the microservices architecture pattern, where each business domain is owned by an independent service with its own database, technology stack, and deployment lifecycle.

The platform demonstrates production-grade engineering practices: idempotent APIs, outbox pattern for reliable messaging, database-per-service isolation, structured JSON logging with distributed tracing via correlation IDs, custom Micrometer metrics exported to Prometheus, and Kubernetes-native health probes.

Core User Journey

User Service
User registers or logs in. Password hashed with BCrypt. JWT-style session stored in browser localStorage.
Catalog Service
Browse events. Each event has metadata — venue, date, category, status (ON_SALE / SOLD_OUT), and pricing tiers.
Seating Service
Interactive seat map renders per event. Seats have statuses: AVAILABLE, HELD, ALLOCATED. A temporary hold (15-min TTL) is placed on selection.
Order Service
Order placed (status: RESERVED). Subtotal + 5% tax calculated. Idempotency key prevents duplicate orders. Outbox event ORDER_PLACED published to RabbitMQ.
Payment Service
Simulated payment gateway processes charge (~98% success). PAYMENT_SUCCESS or PAYMENT_FAILED event published back to RabbitMQ.
Order Service
Consumes payment result. On SUCCESS: order confirmed, e-tickets generated (TKT-* codes). On FAILED: order marked PAYMENT_FAILED.
Notification Service
Consumes ORDER_CONFIRMED event. Sends booking confirmation email (Gmail SMTP or logs if unconfigured). Metric: notifications_sent_total.

Architecture Diagram

Request Data Flow

All external traffic enters through the Nginx Ingress Controller. The React SPA is served by Nginx on port 3001. API calls from the frontend are proxied through Nginx to the appropriate microservice. Inter-service communication uses two patterns:

🔄 Synchronous: OpenFeign (Order→Catalog, Order→Seating) + Axios (Node.js) 📨 Asynchronous: RabbitMQ AMQP (Order↔Payment↔Notification)

Outbox Pattern: Both Order Service and Payment Service use the transactional outbox pattern. Events are written to an outbox table within the same database transaction as the business operation. A scheduler polls and publishes unpublished events to RabbitMQ, guaranteeing at-least-once delivery without distributed transactions.

Connection Pooling

Spring Boot services (order, payment) use HikariCP — maximum 10 connections, minimum 2 idle. Python services use SQLAlchemy's built-in connection pool. Node.js services use mysql2's native pool.

Transactional Outbox Pattern: Business events are committed to an outbox table atomically with domain changes. A scheduled publisher thread reads unpublished rows and delivers them to RabbitMQ, then marks them published. This prevents message loss on service crash between DB commit and broker publish.

Design Patterns & Practices

PatternWhere AppliedPurpose
Database-per-ServiceAll 6 servicesLoose coupling; each service owns its schema and can evolve independently
Transactional OutboxOrder Service, Payment ServiceAtomic event publishing — no dual-write problem with RabbitMQ
Idempotent APIsOrder Service, Payment ServiceIdempotency-Key header prevents duplicate charges/orders on retry
Saga (choreography)Order→Payment→NotificationDistributed transaction across services via event chain, no 2PC
Structured LoggingAll 6 servicesMachine-parseable JSON logs with consistent fields for log aggregation
Correlation ID TracingAll 6 servicesEnd-to-end request correlation across service boundaries via X-Correlation-Id header
Health CheckAll servicesLiveness + readiness probes for Kubernetes to manage pod lifecycle
Sidecar MetricsPrometheus scrapingEach service exposes /metrics; Prometheus scrapes on 15s interval
Seat Hold TTLSeating ServiceTemporary locks prevent double-booking; auto-expire after 15 minutes
Polyglot PersistenceMySQL (4 svc) + PostgreSQL (2 svc)Best-fit database engine per service requirements
API Gateway / IngressNginx Ingress ControllerSingle entry point, TLS termination, routing to services
Multi-stage Docker buildsJava services, FrontendSmaller runtime images (no build tools in final layer)
Polyglot Architecture: SeatSavvy deliberately uses three different backend technology stacks — Python/FastAPI for read-heavy catalog/user operations, Node.js/Express for real-time seat management, and Java/Spring Boot for transactional order and payment processing. Each language is chosen for its strengths in the specific domain.
